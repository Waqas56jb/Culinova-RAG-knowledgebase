The core idea

Two kinds of engineering knowledge, kept separate on purpose

EOS is designed around one principle that keeps it generalized rather than hardcoded: it separates what each equipment category needs from how to size and calculate it. These two layers are authored and stored independently, so either can grow without touching the other — or the code.

Layer A — the knowledge

Engineering Standards Library

Your workbooks. One profile per equipment category, defining every engineering requirement.

  • Required utilities — electrical, water, gas, drainage, ventilation
  • Installation, mechanical & performance characteristics
  • Safety & hygiene requirements, fixed CULINOVA values
  • Which values are manufacturer-dependent vs. rule-driven
Layer B — the calculation

Discipline Rules

The Electrical, Gas & ASHRAE Ventilation rule tables — the engineering math.

  • Condition → output tables (e.g. current band → cable & breaker)
  • Derivation formulas (e.g. current from power, voltage & phase)
  • Calculations (e.g. exhaust airflow by hood grouping)
  • Editable, versioned, engineer-approved — never in code

A Combi Oven standard says: “needs a dedicated circuit, cold water, a Type I hood, 1000 mm clearance — and its cable and breaker come from the Electrical Rules.” The Electrical Rules then turn 12 kW / 400 V into a specific cable and breaker. Standards define the requirement; Rules produce the number. That separation is the whole architecture.

Where the standards live

EOS reads the meaning of every value — not just the text

This is the difference between a spreadsheet and a knowledge base. When a standards workbook is imported, EOS does not store it as a table. It classifies every single cell by what that value tells the engine to do. The eight directive types below are how EOS understands a category from an engineering perspective — and how it knows exactly what it can compute today versus what is waiting on a discipline rule.

Every equipment family workbook is structured slightly differently — different column names, different “not-applicable” markers, extra columns like Data Priority. EOS was built to absorb that variation generically, so the remaining families import with no code change. Today 5 of 9 families are loaded: Cooking · Refrigeration · Food Preparation · Warewashing · Ice Machines — 73 category standards, 4,500 classified data points.

How standards link to equipment

Each machine is bound to its category standard — safely, never guessed

A standard is only useful once it governs a real piece of equipment. EOS matches an equipment item to its category standard by type. An exact match links automatically; a close match is suggested for an engineer to confirm; an uncertain match is never forced. This is already working on live equipment in the system:

The Rules Engine & Dictionary

How EOS calculates, validates, and recommends

The Rules Engine contains no engineering knowledge in code — no cable table, no formula. It only knows how to compare a rule against equipment data and apply it. The Parameter Dictionary is what makes this possible: it turns every way a datasheet spells a value (“Power Load”, “Connected Load”, “Total Power”) into one canonical parameter a rule can reference.

The engine already proves the full electrical chain end-to-end — including deriving a value the datasheet never stated:

Manufacturer data
Phase 3-Phase · Voltage 400 V · Power 12 kW — no current stated (as usual)
Derivation rule
Current = Power × 1000 ÷ (√3 × Voltage × PF) → 19.25 A — formula & power-factor stored as editable engineering data
Recommendation rule
3-Phase, 16–20 A → Cable 5×6 mm² Cu · Breaker 32 A 3P MCB
Traceability
Generated from Electrical Rule E-006 (v1.0) · matched on Phase = 3-Phase, Current 16–20 A · input: derived current 19.25 A

The engine obeys strict policies set by CULINOVA: highest-priority rule wins; two equal rules that disagree raise an engineer conflict — the system never guesses; when no rule matches, the output is left blank and flagged — never extrapolated; and when a rule is updated, every affected item is flagged for recalculation, decided deliberately.

Manufacturer vs. CULINOVA

Two values, side by side — the engineer signs off

EOS never overwrites the manufacturer’s data. Its extracted value and the CULINOVA recommendation are stored together and shown together, so the engineer compares them and decides. Every recommendation carries its full provenance: the rule ID, rule category, rule version, the exact inputs used, the date generated, and who approved it. The engineer can accept, modify, or reject — and a modify or reject requires a written reason. Approval is blocked until every recommendation is resolved. This is what makes EOS an engineering system of record, not a calculator.
